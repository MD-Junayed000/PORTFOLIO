# Metadata stripping and off-topic pronoun refinement for portfolio chatbot

The chatbot was echoing raw PDF metadata (Document ID, Authors, DOI, etc.) directly to users instead of synthesizing natural-language answers. This change rewrites both the pre-LLM context cleaner and the post-LLM response cleaner with broader, more aggressive regex patterns that strip "Key: Value" lines regardless of bullet prefix. It also refines the off-topic detection so bare pronouns ("is he ok?") no longer trick the system into treating a message as portfolio-relevant, and adds a single retry with backoff before falling to fallback responses. `max_new_tokens` drops from 500 to 300.

**Watch for:** The `_clean_context_for_llm` regex `^[\s•◦▪\-*]*[A-Za-z][^:]{0,60}:\s+.+` is aggressive enough to strip legitimate narrative sentences that happen to contain a colon followed by a space (confirmed). The retry adds a fixed 2-second sleep inside the request path with no circuit breaker, so sustained API failures double every request's latency (confirmed). The `_clean_response` bullet-removal regex `^\s*[-*]\s+[^\n]*$` will strip any line starting with a dash or asterisk, which can destroy legitimate LLM prose that begins with "- " for emphasis (likely).

## High-level view

The context cleaner moved from a targeted set of bullet-prefix patterns to a single broad regex that matches any line fitting `Word(s): stuff`. This eliminates metadata echoing but also eliminates any sentence containing a colon followed by content — "His focus: NLP and vision systems" would be silently dropped.

The off-topic detector separates pronouns from substantive keywords. Short messages containing only "he", "his", or "him" without a portfolio keyword are now correctly classified as off-topic, but the 6-word threshold for pronoun-only messages means longer off-topic questions like "does he like what he does?" slip through.

The retry logic wraps the HF API call in a try/catch/sleep/retry before falling back. There is no exponential backoff, no jitter, and no maximum latency cap. Combined with `_call_hf_api`'s internal 503 retry (5s sleep), a single request can block for 17+ seconds before falling back.

The `_clean_response` bullet-removal pass (`^\s*[-*]\s+[^\n]*$`) is indiscriminate — it removes any line starting with dash-space, regardless of whether it's metadata or a valid sentence.

<details>
<summary>Issues (5)</summary>

1. **Over-broad context cleaner regex** — `^[\s•◦▪\-*]*[A-Za-z][^:]{0,60}:\s+.+` matches any line with a word followed by colon-space-content. Legitimate narrative sentences containing colons are silently dropped. Tighten to require known metadata labels or at least require the match to start at column zero with a bullet prefix.
2. **Bullet-stripping removes valid prose** — `^\s*[-*]\s+[^\n]*$` in `_clean_response` removes any line starting with dash-space. If the LLM produces a sentence beginning with "- " (which Mistral-class models do), the entire line disappears. Consider limiting removal to lines that also match a metadata structure.
3. **Fixed 2s retry with no ceiling** — The retry sleeps 2 seconds unconditionally. Under sustained HF API outages, every user request takes ~4s minimum (2s sleep + second timeout) before falling back. Add a circuit breaker or at minimum a configurable timeout budget.
4. **No test for colon-containing narrative surviving the context cleaner** — The test suite verifies metadata is stripped but has no assertion that a line like "His specialty: machine learning and NLP" survives cleaning. This is the primary false-positive risk and should be tested.
5. **Pronoun word-boundary vs substring asymmetry** — Pronouns use `\bhis\b` regex while portfolio keywords use plain `in` substring matching. "history" triggers via substring "his" in the keyword list, not through the pronoun path. No current collision causes a bug, but the asymmetry is fragile for future keyword additions.

## Context cleaner: broad colon-matching regex

The key regex in `_clean_context_for_llm`:

```python
if re.match(r'^[\s•◦▪\-*]*[A-Za-z][^:]{0,60}:\s+.+', stripped):
    continue
```

This matches any line where an alphabetic character is followed by up to 60 non-colon characters, a colon, a space, and then anything. A line like "His research focus: detecting hallucinations in language models" is stripped. The intent is to catch "Authors: Muhammad Junayed" and "DOI: 10.1109/xyz", but the pattern has no anchor to metadata-like structure.

The `_clean_response` function takes the correct approach — it enumerates specific metadata labels. The context cleaner should adopt the same strategy or at minimum require the pre-colon segment to be short (<=3 words) or capitalized in a metadata-like way.

## Retry logic stacks with internal 503 handling

```python
except Exception as e:
    logger.warning(...)
    try:
        await asyncio.sleep(2)
        response_text = await _call_hf_api(prompt)
    except Exception as retry_e:
        response_text = _generate_fallback_response(user_message, context)
```

`_call_hf_api` already retries internally on 503 with a 5-second sleep. A request that hits a 503 internally sleeps 5s, retries, fails, propagates to `generate_response`, sleeps 2s more, calls again, hits another 503, sleeps another 5s. With the httpx timeout at 45s, worst-case latency for a single user request is well over a minute before fallback fires.

## Off-topic pronoun edge case

"does he like what he does?" is 7 words with no portfolio keyword and pronouns present — classified as on-topic by the word-count gate. The LLM will produce a reasonable answer from portfolio context regardless, so the practical impact is low, but it reveals that the heuristic conflates message length with portfolio relevance.

## Test coverage gaps

The new test file covers `_clean_context_for_llm`, `_clean_response`, and `_is_off_topic` with 30+ cases. Not tested: colon-containing narrative surviving the context cleaner (the primary false-positive risk), the retry path in `generate_response`, the interaction between both cleaners running on the same content, and latency behavior under API failure.
